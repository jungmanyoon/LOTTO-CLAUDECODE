# 전수 코드 리뷰 마스터 프롬프트 (로또 예측 시스템)

> 외부 AI(Codex/Gemini/ChatGPT) 또는 Claude에 그대로 복붙해 재사용하는 자기완결형 프롬프트.
> 리뷰 5차원: 계산 / 저장 / 불러오기 / 로직 / 흐름·순서. 작성 2026-06-27.

---

```text
당신은 Python 데이터·ML 시스템을 감사해 온 시니어 코드 감사관이다.
아래 "로또 예측 시스템"을 5개 차원(계산·저장·불러오기·로직·흐름/순서)에 걸쳐 전수 리뷰하고,
증거에 기반한 개선안을 우선순위와 함께 제시하라. 코드를 직접 읽고 file:line 근거를 대라. 추측 금지.

[시스템 개요]
- 목적: 8,145,060개 로또 조합 중 "역사적으로 출현율이 극히 낮은 극단 패턴"을 제거하고,
  남은 풀에서 최종 5세트를 선택하는 시스템이다. 진입점은 main.py.
- 제품 핵심(반드시 정확/견고해야 함): 극단성 풀 생성(ExtremenessScorer/마할라노비스 거리 등)
  -> 목표 풀 크기 K로 축소 -> 5세트 다양성 선택. 이 경로가 사용자에게 나가는 최종 산출물이다.
- 보조 계층(있으면 좋으나 최종 5세트 예측에는 직접 영향 없음): ML(LSTM/Ensemble/Monte Carlo,
  온라인 학습), 레거시 16필터(현재 예측에 미사용), 백테스트, 자동최적화(Optuna/threshold),
  웹 대시보드(Flask), 스케줄러(AutoScheduler).
- 설계 철학(평가 기준으로 삼을 것): "로또는 독립시행이라 무의미하다"는 순수 확률론적 비판은
  적용하지 않는다. 이 시스템은 당첨 확률을 높이는 게 아니라 "같은 예산으로 더 그럴듯한 영역을
  커버"하는 전략이며, 이는 확정된 설계다. 리뷰는 오직 "코드가 이 전략을 정확하게, 정직하게,
  견고하게 구현하는가"로 평가한다.

[리뷰 5차원과 차원별 점검 항목]
1. 계산(정확성·수치안정성)
   - 부동소수점 동치 비교(==), 0 나눗셈, log(0)/sqrt(음수), 오버플로/언더플로
   - 공분산 역행렬 특이/조건수, 마할라노비스 거리 계산, 표준화(scaler) 일관성(학습 vs 예측)
   - 확률/백분위/누적 통계의 경계 처리와 산식 정확성
   - 점수 정규화 공식의 일관성(예: 0~1 / 0~6 / 0~100 혼용으로 인한 비교 오류)
   - 난수 시드와 재현성(같은 입력 -> 같은 결과 보장 지점, 의도적 변동 지점 구분)
2. 저장(DB·직렬화)
   - 원자적 쓰기(임시파일+os.replace), 동시 쓰기 경합, 트랜잭션/락(SQLite WAL, busy_timeout)
   - JSON/DB 스키마 일관성, 직렬화에서 누락된 상태 필드(복원 시 KeyError/기본값 처리)
   - 인덱스 부재로 인한 풀스캔, 무한 증가 테이블(cleanup 호출처 존재 여부), VACUUM
   - 저장 순서 단절로 값이 항상 0/기본값으로 남는 컬럼
   - 인코딩(파일 I/O가 utf-8 명시했는가)
3. 불러오기(상태복원·캐시)
   - 상태파일 구버전 호환(키 누락 시 기본값), 부분 복원/손상 복구
   - 캐시 무효화 트리거(데이터/회차 변경 감지), 캐시 손상 시 자동 재생성
   - 모델 캐시 해시 버전과 사이드카(예: trained_round) 정합, 새 회차 silent 재사용 여부
   - 싱글톤 초기화 순서/재진입, reset 시 모든 관련 경로(메모리+디스크) 무효화
4. 로직(알고리즘 정합성)
   - 극단성 점수 -> 풀 축소 -> 5세트 다양성 선택의 수학적 정합(겹침/커버리지 기준)
   - 조건 경계 버그(이른 break/continue, off-by-one, 누적 카운터 vs 절대값 혼동)
   - 폴백 경로가 가짜/더미 데이터를 만들지 않는가(실데이터 없으면 명시적 실패, NO FAKE DATA)
   - 죽은 코드/no-op(호출처 0, 결과가 영속/사용되지 않는 연산)
5. 흐름·순서(파이프라인·실행시퀀스·동시성)
   - main.py 실행 시퀀스: 데이터수집 -> 필터 -> ML -> 백테스트 -> 최종예측 -> (상주)
   - daemon 스레드 생명주기, graceful shutdown, stop-flag 전파, 종료 시 쓰레기 데이터 방지
   - DB 락/경합, ProcessPool 워커 수/배치, 메모리 경합과 OOM 가드
   - subprocess 자기 재실행(스케줄러)과 중복 실행 방지
   - 데이터 누설(data leakage) 특별 점검: 백테스트/학습이 train_until 이후(미래) 회차를 쓰지 않는가,
     풀 생성이 r <= train_until 범위만 쓰는가

[분석 방법 — 이 순서로 수행]
1. 코드 지도부터 그린다: main.py 진입점에서 출발해 핵심 모듈과 데이터 흐름(입력 DB -> 변환 -> 출력)을 1장으로.
2. 각 차원 체크리스트로 이상 징후를 전방위 탐지한다. 발견마다 실제 파일을 읽고 file:line과 코드 스니펫을 근거로 댄다.
3. 발견마다 "적대적 반증"을 시도한다: 정말 버그인가? 의도된 동작인가? 테스트/호출처/가드가 막아주는가?
   반증에 실패한(=진짜로 문제인) 것만 confirmed로 보고한다. 다중비교 착시·미적용 코드는 배제한다.
4. 우선순위는 영향 경로로 매긴다: 제품 핵심(극단성 풀->5세트) 경로의 결함이 최우선.
   보조 계층은 정확성과 함께 "정직성"(일하는 척하는가)도 본다.
5. 개선안은 외과적(최소 변경)으로 제시하고, 회귀 위험과 검증 방법(어떤 테스트로 확인하는지)을 반드시 동반한다.

[제약]
- 모든 발견은 file:line + 코드 스니펫 근거. 근거 없는 일반론 금지.
- 영향 경로를 명시한다(제품핵심/보조계층/표시계층 중 무엇에 닿는가).
- 제품 핵심 전략 자체를 "확률적으로 무의미"라고 비판하지 않는다(위 설계 철학).
- 수정안은 큰 재작성이 아니라 최소 변경 우선. 새 의존성/대규모 리팩토링은 별도 "구조 제안"으로 분리.

[출력 형식]
각 발견을 다음 구조로 적는다:
- [차원] 제목
- 위치: path/file.py:line
- 증상/근본원인: (무엇이, 왜)
- 판정: real_bug | wrong_behavior | dead_or_dishonest | simplify | ok
- 영향: 제품핵심 | 보조계층 | 표시계층
- 수정안: (file:line, before -> after 요지)
- 리스크: low | med | high + 검증 방법(테스트/실행)
- 우선순위: P0 | P1 | P2 | P3

마지막에:
- 차원별 요약(각 1~2줄)
- 우선순위 Top-N 목록(P0/P1부터)
- 전체 건강도 한 줄 평가(제품 핵심 경로 / 보조 계층 각각)
```
